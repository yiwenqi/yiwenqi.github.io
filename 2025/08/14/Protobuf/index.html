ProtoBuf作为一种跨平台、语言无关、可扩展的序列化数据结构的方法，已经广泛用于网络传输过程中的数据交换与存储；

一、什么是ProtoBuf

  • protobuf 是一种跨语言，跨平台，可扩展的序列化数据结构的方法，用于网络传输过程中的数据交换以及存储；

    在网络传输过程当中除了ProtoBuf 数据结构我们还有json，xml两种高效的数据存储形式，但这两种数据结构的序列号以及反序列化的性能略低于ProtoBuf; 并且使用Protobuf序列化的数据占用空间更小；

二、为什么是选择ProtoBuf

现有框架中除了Protobuf 还有 thrift也可以作为传输层的协议进行数据传输；那为什么选择ProtoBuf? 我们先看ProtoBuf&thrift的区别：

维度 Protobuf 优点 Thrift 优点
序列化性能 非常高效，序列化后数据体积小 性能也很好，但通常略逊于 Protobuf
IDL 语法 简洁，结构清晰（.proto 文件） 功能更丰富，支持异常、默认值、常量等（.thrift 文件）
RPC 支持 原生不支持 RPC，但可与 gRPC 配合实现 自带完整的 RPC 框架，包括传输层、协议层
多语言支持 官方支持主流语言（但冷门语言支持有限） 支持语言更多，适用于异构系统
兼容性 向后兼容好：字段支持 optional 和 repeated 也支持向后兼容，但字段处理更复杂一点
文档生成 官方工具支持较少 工具生态更强，如 Thrift compiler 可自动生成服务端与客户端代码
开发复杂度 比较简单，尤其适合只做数据结构序列化的场景 学习曲线略陡，适合需要完整 RPC 系统的复杂业务场景
生态成熟度 与 gRPC 一起生态非常成熟，广泛用于 Kubernetes、Istio 等云原生技术 生态完善但略显陈旧，新项目采用相对较少

从上述看 ProtoBuf 相比 thrift 虽然单独相比 thrift 的功能更加丰富，但ProtoBuf 与 grpc 配合使用也有丰富强大的功能；且 ProtoBuf 性能更好序列化后的体积更小，生态更加成熟；

四、ProtoBuf 工作原理

​ 序列化/反序列化：把数据对象转换成二进制/把二进制数据转换为数据对象；

​ 序列化&反序列化属于通讯协议的一部分，在通讯协议的通常在TCP/IP模型的应用层，在OSI七层模型的表示层中；

3.1 ProtoBuf 序列号的数据类型

  • Protcol Buffer 对不同的数据类型有不同的序列化方式（编码方式&存储方式）

3.2 Varint 编码方式（Write Type = 1）

​ 使用Varint编码多用于数字类型数据，此时使用变长的编码方式（1-10个字节）；

  • 长度：1-10个字节

  • 原理：根据不同数字类型使用不同的编码方式（直接返回无符号整型或者zigzag编码后的数据）

  • 作用：通过减少表示数字的字节数从而进行数据压缩

    • 对于 int32 类型的数字，一般需要 4个字节 表示； 若采用 Varint编码，对于很小的 int32 类型 数字，则可以用 1个字节 来表示；
    数字 编码后占用字节数（Varint）
    0 ~ 127 1 字节
    128 ~ 16,383 2 字节
    16,384 ~ 2M+ 3 字节
    越大 越多字节

源码分析：

  • 判断v是否小于 1<<7 (127)，1<<14 (16383) …….
  • 每个字节最多存 7 位有效数据（低 7 位），高位（MSB）是继续标志位（1 表示还有更多，0 表示结束）。
  • 这样的编码方式使得数据越小占用空间越小，程序中大部分情况都是小数情况，因此能节约一部分存储空间；
  • 同样这样的编码方式也使得数据计算简单，高性能

3.3 ZigZag编码方式

  • 以上Varint编码在负数时二进制特别大，例如 -300：int(32) => 11111111 11111111 11111110 11010100
  • 因此此时需要用到四个字节；占用空间较大
  • 换成ZigZag后将负数转换为正数；
